Store

Store is a unique centralized JavaScript Object Literal
This tree representation encompass a specific snapshot state of the whole application

Mostly, three kind of data in our state:

  • Domain Data: data that the application needs to show, use, or modify
  • App State: data that is specific to the application's behavior
  • UI State: data that represents how the UI is currently displayed

Example

  • Domain Data: "all of the Todos retrieved from the server"
  • App State: "Todo #5 is currently selected" || "there is a request in progress to fetch Todos"
  • UI State: "The EditTodo modal dialog is currently open"

Store Normalization

Store can be considerer like a database, thus the table are the reducers
=> selection

State needs: SHARI

  • Shared => many components or services
  • Hydrated => persisted & hydrated from Storage
  • Available => re-entering routes
  • Retrieved => retrieved through side-effect
  • Impacted => State that is impacted from other sources

-Hydrated: like userPreference could be a good things
-Availbale: come back to the page data already here
-Retrieved: Unidirectional Data Flow clean/flaten asynchronous complexity

What doesn't belong state

  • Forms
  • Non serializable Data
  • Ephemeral Data

Forms: Cause stateful, self-contained local context => Angular's Form Control aren't serializable & Immutable
Non serializable data: function eval or Circular Object Data Structure, Map or Set => Router State example
Ephemeral data: Concerning state solely owned by one specific component => I know we break Single truth concept
